## Введение: Что происходит с мономорфизмом?

В докладах и статьях о производительности JavaScript часто подчёркивают важность **мономорфного** кода. Однако они обычно не дают ясного объяснения того, что такое мономорфизм/полиморфизм и почему это важно. Даже в докладах экспертов часто звучит что-то вроде «**ОДИН ТИП — ХОРОШО. ДВА ТИПА — ПЛОХО!!!**».

### Терминологическая путаница

Важно понимать, что слово "полиморфизм" перегружено смыслами:

- **В ООП**: Полиморфизм обычно означает субтипизацию и возможность переопределить поведение базового класса
- **В функциональном программировании**: Параметрический полиморфизм (generics)
- **В контексте производительности JS**: **Полиморфизм в точке вызова** (call-site polymorphism)

Именно последний тип полиморфизма влияет на производительность JavaScript и является темой этой статьи.

## Часть 1: Фундаментальные концепции

### 1.1 Динамическая типизация и её цена

JavaScript является **динамически типизированным** языком. Это означает, что тип значения определяется во время выполнения (runtime), а не во время компиляции.

```javascript
// В JavaScript переменная может хранить значения разных типов
let x = 42;        // number
x = "hello";       // string
x = { foo: 1 };    // object
```

**Теоретическая основа**: В статически типизированных языках (C++, Java, Rust) компилятор знает типы всех переменных на этапе компиляции. Это позволяет:

1. Генерировать оптимальный машинный код напрямую
2. Вычислять смещения полей структур/объектов на этапе компиляции
3. Использовать прямую адресацию памяти

В динамических языках эта информация недоступна заранее, что приводит к **runtime overhead**.

### 1.2 Проблема динамического поиска свойств

Рассмотрим простую операцию:

```javascript
function getX(o) {
  return o.x;  // Как VM находит свойство x?
}
```

**Наивная реализация** (согласно спецификации ECMAScript):

```
1. Проверить, есть ли свойство 'x' на самом объекте o
2. Если нет, проверить прототип объекта o
3. Если нет, проверить прототип прототипа
4. Продолжать до Object.prototype
5. Если не найдено нигде, вернуть undefined
```

**Стоимость наивной реализации**:

- **O(n)** сложность, где n — длина цепочки прототипов
- Множественные обращения к памяти
- Невозможность оптимизации компилятором
- Проверка типов и существования на каждом шаге

#### Диаграмма: Наивный поиск свойства

```
Объект o
   |
   v
[Проверить собственные свойства] --нет--> [Проверить __proto__] --нет--> [Проверить __proto__.__proto__] --> ...
   |                                          |
   да                                         да
   |                                          |
   v                                          v
[Вернуть значение]                      [Вернуть значение]
```

### 1.3 Принцип временной локальности (Temporal Locality)

**Теория**: В информатике принцип локальности утверждает, что программы имеют тенденцию обращаться к одним и тем же данным или инструкциям многократно в течение короткого периода времени.

**Применение к JavaScript**:

```javascript
function processArray(arr) {
  for (let i = 0; i < arr.length; i++) {
    console.log(arr[i].x);  // Доступ к .x повторяется много раз
  }
}

processArray([{x: 1}, {x: 2}, {x: 3}]);  // Все объекты имеют одну структуру
```

**Ключевое наблюдение**: Если операция `o.x` видела объекты определённой структуры, вероятно, она увидит похожие объекты в будущем.

Это наблюдение является основой для **inline caching**.

## Часть 2: Inline Caching — Основа оптимизации

### 2.1 Концепция Inline Cache (IC)

**Определение**: Inline Cache — это техника оптимизации, которая кэширует результаты динамического поиска свойств/методов, чтобы избежать повторных дорогостоящих операций поиска.

**Ключевая идея**: Каждая точка доступа к свойству в коде **учится** на объектах, которые она видела, и использует эти знания для оптимизации будущих обращений.

### 2.2 Теоретическая основа IC

Inline caching основан на нескольких фундаментальных принципах:

#### 2.2.1 Мемоизация (Memoization)

IC — это форма мемоизации, где:
- **Ключ кэша**: Форма (структура) объекта
- **Значение кэша**: Путь к свойству (смещение в памяти или getter-функция)

#### 2.2.2 Компромисс пространство/время

```
Время поиска БЕЗ кэша: O(n) × количество обращений
Время поиска С кэшем:   O(1) × количество обращений + O(1) стоимость проверки
Пространство для кэша:  O(k), где k — количество уникальных форм
```

**Выгода**: Когда количество обращений >> количества уникальных форм

#### 2.2.3 Принцип локальности и CPU cache

Inline cache улучшает не только логическую производительность, но и использование CPU cache:

- **Spatial locality**: Данные кэша находятся рядом с кодом
- **Temporal locality**: Горячие пути используют одни и те же кэшированные данные
- **Предсказуемость для CPU**: Монотонный паттерн доступа к памяти

### 2.3 Как работает Inline Cache

#### Шаг 1: Первое выполнение (Uninitialized IC)

```javascript
function f(o) {
  return o.x;  // IC еще не инициализирован
}

f({x: 1});  // Первый вызов
```

**Что происходит**:
1. VM выполняет полный поиск свойства
2. Определяет "форму" объекта (hidden class)
3. Вычисляет путь к свойству x
4. **Сохраняет в кэше**: `форма {x: *} → смещение @12`

#### Шаг 2: Последующие выполнения (Monomorphic IC)

```javascript
f({x: 2});  // Второй вызов
```

**Что происходит**:
1. Быстрая проверка: `форма(o) === кэшированная_форма?`
2. Если да: прямой доступ по смещению (очень быстро!)
3. Если нет: переход к медленному пути

#### Диаграмма: Работа IC

```
┌─────────────────────────────────────────┐
│  Код: o.x                                │
└─────────────────┬───────────────────────┘
                  │
                  v
        ┌─────────────────┐
        │ Проверка кэша    │
        └────┬────────┬────┘
             │        │
      Попадание    Промах
             │        │
             v        v
      ┌──────────┐  ┌─────────────────┐
      │ Быстрый  │  │ Полный поиск +  │
      │ доступ   │  │ обновление кэша │
      │ O(1)     │  │ O(n)            │
      └──────────┘  └─────────────────┘
```

### 2.4 Структура данных IC

Каждый IC можно представить как:

```javascript
// Псевдокод структуры IC
class InlineCache {
  state: 'uninitialized' | 'monomorphic' | 'polymorphic' | 'megamorphic';
  entries: Array<{
    shape: HiddenClass,      // Форма объекта
    handler: Function | int  // Обработчик или смещение
  }>;
  capacity: number;  // Максимальное количество записей
}
```

**Важно**: IC имеет **фиксированную ёмкость** (обычно 4-5 записей для V8).

## Часть 3: Скрытые классы (Hidden Classes / Shapes / Maps)

### 3.1 Концепция скрытых классов

**Проблема**: JavaScript объекты — это словари (hash maps). Доступ к свойству требует хеш-поиска O(log n) или O(1) с коллизиями.

**Решение**: V8 использует **скрытые классы** (hidden classes), чтобы сделать объекты похожими на объекты в C++/Java.

**Определение**: Скрытый класс (также называемый Shape или Map) — это внутренняя структура данных VM, которая описывает:
1. Какие свойства есть у объекта
2. В каком порядке они были добавлены
3. Где они находятся в памяти (смещения)
4. Их атрибуты (writable, enumerable, configurable)

### 3.2 Как создаются скрытые классы

#### Пример: Эволюция скрытого класса

```javascript
const obj = {};           // Shape S0: пустой объект
obj.x = 1;                // Переход: S0 → S1 (добавлено свойство 'x')
obj.y = 2;                // Переход: S1 → S2 (добавлено свойство 'y')
```

#### Диаграмма: Цепочка переходов скрытых классов

```
┌─────────┐    add 'x'    ┌──────────┐    add 'y'    ┌──────────┐
│ Shape S0│───────────────>│ Shape S1 │───────────────>│ Shape S2 │
│ {}      │                │ {x: *}   │                │ {x:*, y:*}│
└─────────┘                └──────────┘                └──────────┘
                              @offset 0                  @offset 4
```

### 3.3 Memory Layout объектов с одним скрытым классом

**Ключевое свойство**: Объекты с одинаковым скрытым классом имеют **идентичное расположение свойств в памяти**.

```javascript
const obj1 = {x: 1, y: 2};
const obj2 = {x: 3, y: 4};
```

**Memory layout**:

```
obj1:                          obj2:
┌──────────────┐              ┌──────────────┐
│ Hidden Class │───┐          │ Hidden Class │───┐
│ pointer  ────┼───┼──────────┼─────────────────┤
├──────────────┤   │          ├──────────────┤   │
│ x: 1   @0    │   │          │ x: 3   @0    │   │
├──────────────┤   │          ├──────────────┤   │
│ y: 2   @4    │   │          │ y: 4   @4    │   │
└──────────────┘   │          └──────────────┘   │
                   │                              │
                   v                              │
              ┌─────────────┐                     │
              │  Shape S2   │<────────────────────┘
              │  {x:@0,y:@4}│
              └─────────────┘
```

**Преимущество**: IC может использовать одну запись кэша для обоих объектов!

### 3.4 Когда создаются разные скрытые классы

#### Случай 1: Разный порядок добавления свойств

```javascript
const obj1 = {};
obj1.x = 1;  // S0 → S1
obj1.y = 2;  // S1 → S2

const obj2 = {};
obj2.y = 1;  // S0 → S3 (другой путь!)
obj2.x = 2;  // S3 → S4
```

**Результат**: `obj1` и `obj2` имеют **разные** скрытые классы, даже если свойства одинаковые!

#### Диаграмма: Разветвление скрытых классов

```
                 ┌──────────┐
            ┌────┤ Shape S0 │────┐
            │    │   {}     │    │
            │    └──────────┘    │
         add 'x'              add 'y'
            │                    │
            v                    v
       ┌──────────┐         ┌──────────┐
       │ Shape S1 │         │ Shape S3 │
       │  {x: *}  │         │  {y: *}  │
       └─────┬────┘         └─────┬────┘
             │                    │
          add 'y'              add 'x'
             │                    │
             v                    v
       ┌──────────┐         ┌──────────┐
       │ Shape S2 │         │ Shape S4 │
       │ {x:*, y:*}│         │ {y:*, x:*}│
       └──────────┘         └──────────┘
```

**Важно**: S2 ≠ S4, даже если набор свойств одинаковый!

#### Случай 2: Удаление свойств

```javascript
const obj1 = {x: 1, y: 2};
const obj2 = {x: 1, y: 2};
delete obj2.y;  // obj2 переходит в другой скрытый класс!
```

**Рекомендация**: Избегайте `delete` для свойств объектов в горячих путях. Вместо этого используйте `obj.prop = undefined` или `obj.prop = null`.

#### Случай 3: Добавление свойств после создания

```javascript
// Хорошо: все объекты имеют одинаковую форму
const objects = [
  {x: 1, y: 2},
  {x: 3, y: 4},
  {x: 5, y: 6}
];

// Плохо: объекты имеют разные формы
const badObjects = [
  {x: 1},
  {x: 2, y: 3},
  {x: 4, y: 5, z: 6}
];
```

## Часть 4: Состояния Inline Cache

### 4.1 Жизненный цикл IC: Состояния

Каждый IC проходит через несколько состояний в процессе выполнения:

```
Uninitialized → Monomorphic → Polymorphic → Megamorphic
```

#### Диаграмма: Машина состояний IC

```
     ┌──────────────────┐
     │ Uninitialized    │  Начальное состояние
     │ (не видел объектов)│
     └────────┬─────────┘
              │ Увидел объект формы A
              v
     ┌──────────────────┐
     │  Monomorphic     │  Оптимальное состояние
     │  Видел: [A]      │  ОДИН ТИП - ХОРОШО!
     └────────┬─────────┘
              │ Увидел объект формы B (B ≠ A)
              v
     ┌──────────────────┐
     │  Polymorphic     │  Приемлемое состояние
     │  Видел: [A, B]   │  Несколько типов
     │         [A,B,C]  │
     │         [A,B,C,D]│
     └────────┬─────────┘
              │ Увидел > 4 разных форм
              v
     ┌──────────────────┐
     │  Megamorphic     │  Худшее состояние
     │  Видел: много    │  Слишком много типов!
     └──────────────────┘
```

### 4.2 Monomorphic (Мономорфное) состояние

**Определение**: IC видел объекты только **одной** формы.

```javascript
function f(o) {
  return o.x;  // IC для .x
}

// Все вызовы с объектами одной формы
f({x: 1});
f({x: 2});
f({x: 3});

// IC остается мономорфным: видел только форму {x: *}
```

**Характеристики производительности**:
- **Проверка**: Одна проверка указателя на скрытый класс
- **Доступ**: Прямое чтение по смещению
- **Стоимость**: ~1-2 CPU цикла (после inline)
- **Предсказуемость ветвлений**: Отличная (всегда true)

**Псевдокод сгенерированного кода**:

```cpp
// Мономорфный доступ к o.x
if (o.hidden_class == cached_shape_A) {
  return *(o.data + offset_12);  // Прямое чтение
} else {
  deoptimize();  // Редкий путь
}
```

### 4.3 Polymorphic (Полиморфное) состояние

**Определение**: IC видел объекты **2-4 разных форм** (в V8).

```javascript
function f(o) {
  return o.x;  // IC для .x
}

f({x: 1});           // Форма A: {x: *}
f({x: 2, y: 3});     // Форма B: {x: *, y: *}
f({x: 4, y: 5, z: 6}); // Форма C: {x: *, y: *, z: *}

// IC теперь полиморфный степени 3
```

**Характеристики производительности**:
- **Проверка**: Линейный поиск по N формам
- **Доступ**: Прямое чтение по смещению (после нахождения формы)
- **Стоимость**: ~N проверок + доступ
- **Предсказуемость ветвлений**: Зависит от распределения типов

**Псевдокод сгенерированного кода**:

```cpp
// Полиморфный доступ к o.x (3 формы)
if (o.hidden_class == shape_A) {
  return *(o.data + offset_A);
} else if (o.hidden_class == shape_B) {
  return *(o.data + offset_B);
} else if (o.hidden_class == shape_C) {
  return *(o.data + offset_C);
} else {
  deoptimize();
}
```

#### Два вида полиморфных IC в V8

##### 4.3.1 Полиморфный с одинаковыми смещениями

Если свойство находится на **одном и том же смещении** во всех формах:

```javascript
const obj1 = {x: 1};           // x на смещении @0
const obj2 = {x: 2, y: 3};     // x на смещении @0
const obj3 = {x: 4, y: 5, z: 6}; // x на смещении @0
```

**Оптимизация**: V8 может сгенерировать **polymorphic type guard**:

```cpp
// Оптимизированный полиморфный доступ
if (o.hidden_class == shape_A ||
    o.hidden_class == shape_B ||
    o.hidden_class == shape_C) {
  return *(o.data + offset_0);  // Одно смещение для всех!
} else {
  deoptimize();
}
```

**Преимущество**: Только одно чтение памяти после проверки типа.

##### 4.3.2 Полиморфный с разными смещениями

Если свойство на **разных смещениях**:

```javascript
const obj1 = {x: 1};           // x на смещении @0
const obj2 = {y: 2, x: 3};     // x на смещении @4 (после y)
const obj3 = {z: 4, y: 5, x: 6}; // x на смещении @8
```

**Генерируется**: Полное дерево решений (как показано выше).

**Недостаток**: Хуже для CPU branch predictor.

### 4.4 Megamorphic (Мегаморфное) состояние

**Определение**: IC видел объекты **более 4 разных форм** (порог зависит от VM).

```javascript
function f(o) {
  return o.x;
}

// Вызовы с множеством разных форм
f({x: 1});
f({x: 1, a: 2});
f({x: 1, b: 2});
f({x: 1, c: 2});
f({x: 1, d: 2});
f({x: 1, e: 2});  // Переход в мегаморфное состояние!
```

**Характеристики производительности**:
- **Локальный кэш**: Больше не используется
- **Глобальный кэш**: Используется глобальная хеш-таблица (megamorphic stub cache)
- **Стоимость**: Hash lookup + возможные коллизии
- **Предсказуемость**: Плохая

**Глобальная хеш-таблица**:
- Фиксированный размер (например, 1024 записи)
- Общая для всех мегаморфных IC в программе
- Коллизии приводят к перезаписи

**Псевдокод**:

```cpp
// Мегаморфный доступ
Shape shape = o.hidden_class;
Handler handler = global_megamorphic_cache.lookup(shape, "x");
if (handler != null) {
  return handler(o);  // Может быть функция или смещение
} else {
  // Полный runtime lookup
  return runtime_get_property(o, "x");
}
```

### 4.5 Сравнение производительности состояний

#### Таблица: Характеристики производительности

| Состояние | Проверка | Доступ | Относительная скорость | Branch prediction |
|-----------|----------|--------|------------------------|-------------------|
| **Monomorphic** | 1 сравнение | Прямое | **1.0x** (базовая) | Отлично |
| **Polymorphic (2)** | 2 сравнения | Прямое | **0.95x** | Хорошо |
| **Polymorphic (4)** | 4 сравнения | Прямое | **0.85x** | Приемлемо |
| **Polymorphic (разные смещения)** | 4 сравнения + 4 чтения | Условное | **0.70x** | Плохо |
| **Megamorphic** | Hash lookup | Прямое/вызов | **0.30-0.50x** | Очень плохо |

**Примечание**: Точные цифры зависят от паттернов использования, распределения типов и архитектуры CPU.

### 4.6 Практический пример: Визуализация состояний IC

```javascript
// Пример 1: Мономорфный доступ
function processMonomorphic(objects) {
  let sum = 0;
  for (let i = 0; i < objects.length; i++) {
    sum += objects[i].value;  // IC: Monomorphic
  }
  return sum;
}

// Все объекты одной формы
const monoObjects = [
  {value: 1},
  {value: 2},
  {value: 3}
];

// Пример 2: Полиморфный доступ
function processPolymorphic(objects) {
  let sum = 0;
  for (let i = 0; i < objects.length; i++) {
    sum += objects[i].value;  // IC: Polymorphic (степень 3)
  }
  return sum;
}

// Объекты разных форм
const polyObjects = [
  {value: 1},           // Форма A
  {value: 2, x: 0},     // Форма B
  {value: 3, y: 0},     // Форма C
  {value: 1},           // Форма A
  {value: 2, x: 0},     // Форма B
];

// Пример 3: Мегаморфный доступ
function processMegamorphic(objects) {
  let sum = 0;
  for (let i = 0; i < objects.length; i++) {
    sum += objects[i].value;  // IC: Megamorphic
  }
  return sum;
}

// Объекты множества разных форм
const megaObjects = [
  {value: 1},
  {value: 2, a: 0},
  {value: 3, b: 0},
  {value: 4, c: 0},
  {value: 5, d: 0},
  {value: 6, e: 0},  // Более 4 форм!
];
```

**Бенчмарк результаты** (относительные):
- `processMonomorphic`: **1.00x** (базовая скорость)
- `processPolymorphic`: **0.85x** (на 15% медленнее)
- `processMegamorphic`: **0.45x** (более чем в 2 раза медленнее)

## Часть 5: Оптимизирующий компилятор и спекулятивная оптимизация

### 5.1 Архитектура многоуровневого выполнения

Современные JavaScript VM (V8, SpiderMonkey, JavaScriptCore) используют **многоуровневую архитектуру**:

```
┌─────────────────────────────────────────────────────────┐
│                    JavaScript Source                    │
└────────────────────────┬────────────────────────────────┘
                         │
                         v
              ┌──────────────────────┐
              │   Parser + AST       │
              └──────────┬───────────┘
                         │
                         v
              ┌──────────────────────┐
              │  Ignition Bytecode   │  Уровень 1: Интерпретатор
              │  (быстрая компиляция)│  - Низкая задержка запуска
              └──────────┬───────────┘  - Собирает type feedback
                         │
                         │ Горячий код (hot)
                         │
                         v
              ┌──────────────────────┐
              │    TurboFan JIT      │  Уровень 2: Оптимизирующий компилятор
              │  (медленная компиляция│  - Использует type feedback
              │   оптимизированный    │  - Генерирует специализированный код
              │   машинный код)       │  - Может деоптимизировать
              └───────────────────────┘
```

### 5.2 Роль Type Feedback

**Проблема**: JavaScript не имеет статической типизации, но процессору нужен специализированный код.

**Решение**: Собирать **обратную связь по типам** (type feedback) во время выполнения.

**Источники type feedback**:
1. **Inline caches**: Какие формы видел каждый доступ к свойству
2. **Arithmetic ICs**: Какие типы чисел (Smi, HeapNumber) видели арифметические операции
3. **Call ICs**: Какие функции вызывались в точке вызова
4. **Comparison ICs**: Какие типы сравнивались

#### Пример: Type Feedback в действии

```javascript
function add(a, b) {
  return a + b;  // Что такое a и b?
}

// Фаза 1: Сбор feedback (интерпретатор)
add(1, 2);        // Feedback: a=Smi, b=Smi → Smi
add(3, 4);        // Feedback: подтверждено
add(5, 6);        // Feedback: подтверждено

// Фаза 2: Оптимизация (TurboFan)
// Компилятор видит feedback: "всегда small integers"
// Генерирует специализированный код для Smi сложения
```

### 5.3 Спекулятивная оптимизация

**Определение**: Оптимизирующий компилятор делает **предположения** (speculations) о типах и поведении кода на основе type feedback.

**Процесс**:
1. Анализ type feedback из IC
2. Генерация **специализированного** IR (Intermediate Representation)
3. Добавление **type guards** для проверки предположений
4. Оптимизация IR (устранение избыточности, инлайнинг, и т.д.)
5. Генерация машинного кода

#### 5.3.1 Пример: Оптимизация мономорфного доступа

**Исходный код**:

```javascript
function distance(o) {
  return Math.sqrt(o.x * o.x + o.y * o.y);
}
```

**Type feedback** (после прогрева):
- `o` всегда имеет форму `{x: number, y: number}`
- `o.x` всегда на смещении @0
- `o.y` всегда на смещении @8

**Начальный IR** (упрощённый):

```
// Неоптимизированный IR
v0 = Parameter(0)                    // o
CheckMap v0, Shape{x,y}              // Проверка формы
v1 = LoadField v0, @0                // o.x
CheckMap v0, Shape{x,y}              // Проверка формы (избыточная!)
v2 = LoadField v0, @0                // o.x (избыточная загрузка!)
v3 = Float64Mul v1, v2               // o.x * o.x

CheckMap v0, Shape{x,y}              // Проверка формы (избыточная!)
v4 = LoadField v0, @8                // o.y
CheckMap v0, Shape{x,y}              // Проверка формы (избыточная!)
v5 = LoadField v0, @8                // o.y (избыточная загрузка!)
v6 = Float64Mul v4, v5               // o.y * o.y

v7 = Float64Add v3, v6               // o.x * o.x + o.y * o.y
v8 = CallBuiltin[Math.sqrt], v7     // Math.sqrt(...)
Return v8
```

**Проблемы**:
- 4 избыточные проверки формы
- 2 избыточные загрузки

**После оптимизации (GVN - Global Value Numbering)**:

```
// Оптимизированный IR
v0 = Parameter(0)                    // o
CheckMap v0, Shape{x,y}              // Одна проверка!
v1 = LoadField v0, @0                // o.x (одна загрузка)
v3 = Float64Mul v1, v1               // o.x * o.x

v4 = LoadField v0, @8                // o.y (одна загрузка)
v6 = Float64Mul v4, v4               // o.y * o.y

v7 = Float64Add v3, v6               // o.x * o.x + o.y * o.y
v8 = CallBuiltin[Math.sqrt], v7     // Math.sqrt(...)
Return v8
```

**Выигрыш**:
- Устранены 3 проверки формы
- Устранены 2 загрузки полей
- Код более компактный и быстрый

### 5.4 Деоптимизация (Deoptimization)

**Проблема**: Что если наши предположения неверны?

**Решение**: **Деоптимизация** — переход обратно к неоптимизированному коду.

#### Когда происходит деоптимизация:

1. **Type guard failure**: Объект не той формы, что ожидалось
2. **Integer overflow**: Результат не помещается в Smi
3. **Array bounds**: Доступ за границами массива
4. **Prototype chain change**: Изменилась цепочка прототипов
5. **Global property change**: Изменилось глобальное свойство, на которое полагалась оптимизация

#### Пример деоптимизации:

```javascript
function process(o) {
  return o.x + o.y;
}

// Прогрев: все объекты формы {x: Smi, y: Smi}
for (let i = 0; i < 10000; i++) {
  process({x: i, y: i * 2});
}

// TurboFan оптимизировал для Smi сложения

// Деоптимизация!
process({x: 1.5, y: 2.5});  // Теперь HeapNumber, не Smi!
```

**Что происходит**:
1. Type guard проверяет тип `o.x` → ожидает Smi, получает HeapNumber
2. Деоптимизация: возврат к Ignition bytecode
3. Продолжение выполнения в интерпретаторе
4. Сбор нового feedback
5. Возможная ре-оптимизация с новыми предположениями

#### Диаграмма: Цикл оптимизации/деоптимизации

```
┌────────────────┐
│   Bytecode     │
│ (Интерпретатор)│
└───────┬────────┘
        │
        │ Hot code
        │
        v
┌────────────────┐
│  Оптимизация   │
│  (TurboFan)    │
└───────┬────────┘
        │
        v
┌────────────────┐     Предположение      ┌────────────────┐
│ Оптимизированный│────нарушено!──────────>│ Деоптимизация  │
│  машинный код  │                        └───────┬────────┘
└────────────────┘                                │
        ^                                         │
        │                                         │
        │        Новый feedback                   │
        │        + ре-оптимизация                 │
        └─────────────────────────────────────────┘
```

**Важно**: Частая деоптимизация очень вредна для производительности!

### 5.5 Оптимизация полиморфного кода

#### 5.5.1 Полиморфный доступ с деревом решений

Когда IC полиморфен, оптимизатор строит **дерево решений** (decision tree):

```javascript
function get(o) {
  return o.x;
}

// IC видел 3 формы: A, B, C
```

**Сгенерированный оптимизированный код**:

```cpp
// Дерево решений для полиморфного доступа
if (o.map == MapA) {
  return load_field(o, offsetA);
} else if (o.map == MapB) {
  return load_field(o, offsetB);
} else if (o.map == MapC) {
  return load_field(o, offsetC);
} else {
  deoptimize();  // Увидели новую форму!
}
```

**Проблемы**:
- Множественные ветвления
- Плохо для CPU branch predictor
- Затрудняет дальнейшие оптимизации

#### 5.5.2 Polymorphic Type Guard (оптимизация)

Если свойство на **одном смещении** во всех формах:

```cpp
// Оптимизированный полиморфный guard
if (o.map == MapA || o.map == MapB || o.map == MapC) {
  return load_field(o, offset);  // Одно смещение!
} else {
  deoptimize();
}
```

**Преимущество**: Меньше обращений к памяти.

#### 5.5.3 Влияние на последующие оптимизации

**Проблема**: Полиморфные операции затрудняют устранение избыточности.

```javascript
function process(o) {
  const a = o.x;  // Полиморфный доступ
  const b = o.x;  // Можем ли устранить эту загрузку?
  return a + b;
}
```

**Мономорфный случай**: После первого `o.x` мы знаем, что `o` имеет определённую форму. Вторая загрузка избыточна!

**Полиморфный случай**: После первого `o.x` мы знаем только, что форма — одна из `{A, B, C}`. Это слишком слабая информация для устранения второй загрузки.

### 5.6 Инлайнинг (Inlining) и полиморфизм

**Инлайнинг** — одна из важнейших оптимизаций. Он позволяет:
- Устранить overhead вызова функции
- Оптимизировать код функции в контексте вызывающего кода
- Проводить межпроцедурные оптимизации

#### Мономорфные вызовы

```javascript
class Point {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }

  distance() {
    return Math.sqrt(this.x ** 2 + this.y ** 2);
  }
}

function processPoints(points) {
  let sum = 0;
  for (let p of points) {
    sum += p.distance();  // Мономорфный вызов
  }
  return sum;
}
```

**Оптимизация**: TurboFan может **заинлайнить** `distance()`:

```javascript
// Оптимизированный эквивалент (концептуально)
function processPoints_optimized(points) {
  let sum = 0;
  for (let p of points) {
    // Инлайнинг distance()
    const temp = Math.sqrt(p.x ** 2 + p.y ** 2);
    sum += temp;
  }
  return sum;
}
```

#### Полиморфные вызовы

```javascript
function processShapes(shapes) {
  let sum = 0;
  for (let s of shapes) {
    sum += s.area();  // Полиморфный вызов!
  }
  return sum;
}

// shapes содержит Circle, Rectangle, Triangle
```

**Оптимизация**: TurboFan строит дерево решений с инлайнингом:

```cpp
// Концептуальный оптимизированный код
for (s in shapes) {
  if (s.map == CircleMap) {
    // Инлайн Circle.area()
    sum += PI * s.radius * s.radius;
  } else if (s.map == RectangleMap) {
    // Инлайн Rectangle.area()
    sum += s.width * s.height;
  } else if (s.map == TriangleMap) {
    // Инлайн Triangle.area()
    sum += 0.5 * s.base * s.height;
  } else {
    deoptimize();
  }
}
```

**Проблема**: Код становится больше, сложнее для CPU cache.

## Часть 6: Влияние на производительность процессора

### 6.1 CPU Pipeline и Branch Prediction

Современные процессоры используют **конвейерную обработку** (pipelining) для параллельного выполнения инструкций.

#### Упрощённый CPU Pipeline:

```
Fetch → Decode → Execute → Memory → Writeback
  ↓       ↓        ↓         ↓         ↓
 inst1   inst1    inst1     inst1     inst1
 inst2   inst2    inst2     inst2     inst2
 inst3   inst3    inst3     inst3     inst3
```

**Проблема**: Условные переходы (branches) нарушают pipeline.

#### Branch Prediction

**Определение**: CPU пытается предсказать, какой путь выберет условный переход.

**Правильное предсказание**: Pipeline продолжает работу.
**Неправильное предсказание**: Pipeline flush → потеря ~10-20 циклов!

#### Мономорфный код и CPU

```cpp
// Мономорфный: одна проверка, всегда true
if (o.map == expected_map) {  // Предсказуемо!
  return load_field(o, offset);
}
```

**Branch predictor**: Быстро учится, что ветвь всегда true → 99%+ accuracy.

#### Полиморфный код и CPU

```cpp
// Полиморфный: множественные ветвления
if (o.map == MapA) {       // 40% true
  return load(o, offset_a);
} else if (o.map == MapB) { // 35% true
  return load(o, offset_b);
} else if (o.map == MapC) { // 25% true
  return load(o, offset_c);
}
```

**Branch predictor**: Сложнее предсказать → больше mispredictions → медленнее.

### 6.2 CPU Cache и Memory Locality

#### Cache Hierarchy

```
┌─────────────────────────────────────┐
│ CPU Registers (1 cycle)             │
├─────────────────────────────────────┤
│ L1 Cache (32-64KB, 3-4 cycles)      │
├─────────────────────────────────────┤
│ L2 Cache (256-512KB, 10-20 cycles)  │
├─────────────────────────────────────┤
│ L3 Cache (2-16MB, 40-75 cycles)     │
├─────────────────────────────────────┤
│ RAM (GB, 100-300 cycles)            │
└─────────────────────────────────────┘
```

#### Мономорфный код → Лучшая локальность

```javascript
// Мономорфный: все объекты имеют одинаковый layout
const points = [
  {x: 1, y: 2},  // Memory layout: [map_ptr, x, y]
  {x: 3, y: 4},  // Memory layout: [map_ptr, x, y]
  {x: 5, y: 6},  // Memory layout: [map_ptr, x, y]
];

// Доступ к .x предсказуем → хорошо для prefetcher
for (let p of points) {
  sum += p.x;
}
```

**CPU prefetcher**: Может предзагружать следующие элементы в cache.

#### Полиморфный код → Плохая локальность

```javascript
// Полиморфный: разные layouts
const mixed = [
  {x: 1},           // Layout: [map_ptr, x]
  {x: 2, y: 3},     // Layout: [map_ptr, x, y]
  {y: 4, x: 5},     // Layout: [map_ptr, y, x]  (разный порядок!)
];

// Доступ к .x непредсказуем → плохо для prefetcher
for (let obj of mixed) {
  sum += obj.x;
}
```

**CPU prefetcher**: Не может эффективно предсказать паттерн → больше cache misses.

### 6.3 Instruction Cache (I-Cache)

**Проблема с полиморфным кодом**: Больше машинного кода.

```
Мономорфный:      ~10-20 инструкций
Полиморфный (4):  ~40-80 инструкций
Мегаморфный:      ~100+ инструкций + вызов
```

**Влияние на I-cache**:
- Мономорфный код помещается в L1 I-cache
- Полиморфный код может переполнить L1 → L2 cache misses
- Мегаморфный код + множественные функции → еще хуже

## Часть 7: Практические рекомендации

### 7.1 Как писать мономорфный код

#### Правило 1: Используйте фабричные функции / классы

**Плохо**:

```javascript
// Объекты создаются по-разному
function createPoint1() {
  const p = {};
  p.x = 0;
  p.y = 0;
  return p;
}

function createPoint2() {
  return { x: 0, y: 0 };
}

// Разные формы!
const p1 = createPoint1();
const p2 = createPoint2();
```

**Хорошо**:

```javascript
// Класс гарантирует одинаковую форму
class Point {
  constructor(x = 0, y = 0) {
    this.x = x;
    this.y = y;
  }
}

// Все точки имеют одну форму!
const p1 = new Point();
const p2 = new Point(1, 2);
```

#### Правило 2: Инициализируйте все свойства в конструкторе

**Плохо**:

```javascript
class Entity {
  constructor(x, y) {
    this.x = x;
    this.y = y;
    // health не инициализирован!
  }

  spawn() {
    this.health = 100;  // Изменение формы!
  }
}
```

**Хорошо**:

```javascript
class Entity {
  constructor(x, y) {
    this.x = x;
    this.y = y;
    this.health = 0;  // Инициализируем сразу
  }

  spawn() {
    this.health = 100;  // Просто изменение значения
  }
}
```

#### Правило 3: Инициализируйте свойства в одном порядке

**Плохо**:

```javascript
function createUser(name, age) {
  if (age !== undefined) {
    return { name, age };  // Форма A: {name, age}
  } else {
    return { name };        // Форма B: {name}
  }
}
```

**Хорошо**:

```javascript
function createUser(name, age = null) {
  return { name, age };  // Всегда форма {name, age}
}
```

#### Правило 4: Избегайте delete

**Плохо**:

```javascript
const obj = { x: 1, y: 2, temp: 3 };
// ... использование ...
delete obj.temp;  // Изменение формы!
```

**Хорошо**:

```javascript
const obj = { x: 1, y: 2, temp: 3 };
// ... использование ...
obj.temp = null;  // Форма не изменяется
```

#### Правило 5: Используйте TypeScript/Flow

TypeScript помогает поддерживать согласованные формы:

```typescript
interface Point {
  x: number;
  y: number;
}

// Компилятор гарантирует, что все Point имеют одну форму
function process(p: Point) {
  return p.x + p.y;
}
```

### 7.2 Когда полиморфизм приемлем

**Важно**: Не весь полиморфизм плох! Следуйте правилу **80/20**:

#### Полиморфизм в холодном коде — ОК

```javascript
// Выполняется один раз при загрузке
function initializePlugins(plugins) {
  for (let plugin of plugins) {
    plugin.init();  // Полиморфный вызов — ОК!
  }
}
```

**Причина**: Не в горячем пути, производительность не критична.

#### Умеренный полиморфизм (2-3 типа) — Обычно ОК

```javascript
// Обработка разных типов событий
function handleEvent(event) {
  switch (event.type) {  // 3-4 типа событий
    case 'click': return handleClick(event);
    case 'keypress': return handleKeypress(event);
    case 'scroll': return handleScroll(event);
  }
}
```

**Причина**: V8 хорошо оптимизирует полиморфизм степени 2-4.

#### Абстракция важнее производительности

```javascript
// Хорошая архитектура с полиморфизмом
interface Shape {
  area(): number;
}

class Circle implements Shape { ... }
class Rectangle implements Shape { ... }

function totalArea(shapes: Shape[]) {
  return shapes.reduce((sum, s) => sum + s.area(), 0);
}
```

**Когда оптимизировать**: Только если профилирование показало bottleneck.

### 7.3 Как обнаружить проблемы с полиморфизмом

#### Метод 1: V8 Runtime Flags

```bash
# Вывести состояние IC
node --trace-ic your-script.js

# Вывести деоптимизации
node --trace-deopt your-script.js

# Вывести оптимизации
node --trace-opt your-script.js
```

**Пример вывода**:

```
[LoadIC in ~process at offset 42 (0x...) ((PREMONOMORPHIC|MONOMORPHIC|POLYMORPHIC|MEGAMORPHIC))]
```

#### Метод 2: Chrome DevTools

1. Откройте Chrome DevTools
2. Performance tab → Record
3. Анализируйте профиль:
   - Смотрите на функции с высоким самым временем (self time)
   - Проверяйте "Not optimized" в bottom-up view

#### Метод 3: Turbolizer (для экспертов)

Turbolizer — инструмент для визуализации IR TurboFan:

```bash
# Генерация trace файла
node --trace-turbo your-script.js

# Открыть в Turbolizer
# https://v8.github.io/tools/head/turbolizer/index.html
```

**Что искать**:
- Множественные `CheckMaps` для одного значения
- `Generic` операции в горячих циклах
- Деревья решений с множественными ветвями

### 7.4 Паттерны оптимизации

#### Паттерн 1: Мономорфизация через разделение

**Проблема**: Одна функция обрабатывает разные типы.

```javascript
function process(items) {
  for (let item of items) {
    console.log(item.value);  // Полиморфный!
  }
}

// items содержит TypeA и TypeB
```

**Решение**: Разделите на мономорфные функции.

```javascript
function processTypeA(items) {
  for (let item of items) {
    console.log(item.value);  // Мономорфный для TypeA
  }
}

function processTypeB(items) {
  for (let item of items) {
    console.log(item.value);  // Мономорфный для TypeB
  }
}

function process(items) {
  const typeA = items.filter(isTypeA);
  const typeB = items.filter(isTypeB);
  processTypeA(typeA);
  processTypeB(typeB);
}
```

#### Паттерн 2: Структуры данных по типам

**Проблема**: Смешанные типы в одном массиве.

```javascript
const entities = [
  new Player(),
  new Enemy(),
  new Player(),
  new Bullet(),
];

// Полиморфная итерация
for (let e of entities) {
  e.update();
}
```

**Решение**: Разделите по типам.

```javascript
const game = {
  players: [],  // Только Player
  enemies: [],  // Только Enemy
  bullets: [],  // Только Bullet
};

// Мономорфные итерации
for (let p of game.players) p.update();
for (let e of game.enemies) e.update();
for (let b of game.bullets) b.update();
```

#### Паттерн 3: Тегированные объединения (Tagged Unions)

Для случаев, где нужен полиморфизм:

```javascript
// Вместо разных классов используйте теги
const shapes = [
  { type: 'circle', radius: 5 },
  { type: 'rectangle', width: 10, height: 20 },
];

function area(shape) {
  // Мономорфный доступ к .type
  switch (shape.type) {
    case 'circle':
      return Math.PI * shape.radius ** 2;
    case 'rectangle':
      return shape.width * shape.height;
  }
}
```

**Преимущество**: Все объекты могут иметь базовую форму `{type: string, ...}`.

## Часть 8: Расширенные темы

### 8.1 Прототипная цепочка и производительность

**Проблема**: Свойства в прототипе требуют дополнительных проверок.

```javascript
class Base {
  constructor() {
    this.x = 0;
  }

  method() { return this.x; }
}

class Derived extends Base {
  constructor() {
    super();
    this.y = 0;
  }
}

const obj = new Derived();
obj.method();  // Метод в прототипе Base.prototype
```

**IC должен проверить**:
1. `obj` не имеет собственного свойства `method`
2. `Derived.prototype` не имеет `method`
3. `Base.prototype` имеет `method`

**Оптимизация V8**: Проверка всей цепочки кэшируется в IC.

### 8.2 Геттеры и сеттеры

Геттеры/сеттеры требуют вызовов функций:

```javascript
class Point {
  constructor(x, y) {
    this._x = x;
    this._y = y;
  }

  get x() { return this._x; }
  set x(value) { this._x = value; }
}

const p = new Point(1, 2);
const x = p.x;  // Вызов getter функции!
```

**Стоимость**: Выше, чем прямой доступ к полю.

**Оптимизация**: TurboFan может инлайнить простые геттеры/сеттеры.

### 8.3 Proxy и производительность

**Proxy** — мощный механизм перехвата операций, но очень дорогой:

```javascript
const target = { x: 1, y: 2 };
const proxy = new Proxy(target, {
  get(obj, prop) {
    console.log(`Getting ${prop}`);
    return obj[prop];
  }
});

proxy.x;  // Всегда вызов trap функции!
```

**Проблема**: Proxy операции не могут быть оптимизированы IC.

**Рекомендация**: Избегайте Proxy в горячих путях.

### 8.4 Массивы и элементы разных типов

V8 использует разные внутренние представления для массивов:

```javascript
// PACKED_SMI_ELEMENTS - быстрый
const arr1 = [1, 2, 3, 4, 5];

// PACKED_DOUBLE_ELEMENTS - быстрый
const arr2 = [1.1, 2.2, 3.3];

// PACKED_ELEMENTS - медленнее
const arr3 = [1, 'hello', {}, null];

// HOLEY_ELEMENTS - еще медленнее
const arr4 = [1, , , 4];  // holes
```

**Рекомендация**: Используйте массивы одного типа.

## Часть 9: Глоссарий терминов

### A-D

**Baseline Compiler (Базовый компилятор)**
- **Русский**: Базовый компилятор, неоптимизирующий компилятор
- **Определение**: Компилятор, который быстро генерирует простой машинный код или байткод без оптимизаций. В V8 — Ignition.
- **Цель**: Быстрый запуск, низкая задержка компиляции

**Branch Prediction (Предсказание ветвлений)**
- **Русский**: Предсказание переходов
- **Определение**: Механизм CPU, который пытается предсказать результат условных переходов для поддержания pipeline.
- **Влияние**: Мономорфный код → высокая точность предсказаний

**Call-site Polymorphism (Полиморфизм в точке вызова)**
- **Русский**: Полиморфизм в месте вызова
- **Определение**: Ситуация, когда одна точка в коде обрабатывает значения разных типов/форм.
- **Пример**: `function f(o) { return o.x; }` вызывается с объектами разных форм

**Deoptimization (Деоптимизация)**
- **Русский**: Деоптимизация, откат оптимизации
- **Определение**: Процесс возврата от оптимизированного кода к неоптимизированному, когда предположения оптимизатора нарушены.
- **Причины**: Неожиданный тип, изменение формы, переполнение

### E-H

**Feedback Vector (Вектор обратной связи)**
- **Русский**: Вектор обратной связи по типам
- **Определение**: Структура данных, хранящая информацию о типах и формах, наблюдаемых во время выполнения.
- **Использование**: Оптимизирующий компилятор читает feedback для специализации кода

**GVN (Global Value Numbering)**
- **Русский**: Глобальная нумерация значений
- **Определение**: Оптимизация компилятора, которая устраняет избыточные вычисления.
- **Пример**: Две загрузки `o.x` → одна загрузка + переиспользование

**Hidden Class / Shape / Map (Скрытый класс / Форма / Карта)**
- **Русский**: Скрытый класс, форма объекта
- **Определение**: Внутренняя структура данных VM, описывающая структуру объекта.
- **Содержит**: Список свойств, их смещения, атрибуты
- **Аналог**: Класс в C++/Java

**Hot Code / Hot Path (Горячий код / Горячий путь)**
- **Русский**: Горячий код, часто выполняемый код
- **Определение**: Код, который выполняется очень часто (в циклах, частые функции).
- **Значение**: Кандидат для оптимизации компилятором

### I-L

**Inline Cache (IC)**
- **Русский**: Встроенный кэш, инлайн-кэш
- **Определение**: Механизм кэширования результатов динамических операций (поиск свойств, вызовы).
- **Ключ**: Форма объекта
- **Значение**: Путь к свойству или обработчик

**Inlining (Инлайнинг, встраивание)**
- **Русский**: Встраивание, инлайнинг
- **Определение**: Оптимизация, при которой код вызываемой функции вставляется непосредственно в место вызова.
- **Преимущество**: Устраняет overhead вызова, позволяет межпроцедурные оптимизации

**Intermediate Representation (IR)**
- **Русский**: Промежуточное представление
- **Определение**: Внутреннее представление кода в оптимизирующем компиляторе, более низкоуровневое чем источник.
- **Цель**: Удобно для анализа и оптимизации

### M-P

**Megamorphic (Мегаморфный)**
- **Русский**: Мегаморфный, сверхполиморфный
- **Определение**: Состояние IC, когда он видел больше типов, чем может локально кэшировать (>4 в V8).
- **Производительность**: Худшая среди IC состояний
- **Механизм**: Использует глобальную хеш-таблицу

**Monomorphic (Мономорфный)**
- **Русский**: Мономорфный, однотипный
- **Определение**: Состояние IC, когда он видел объекты только одной формы.
- **Производительность**: Лучшая, оптимальная
- **Лозунг**: "ОДИН ТИП — ХОРОШО"

**Polymorphic (Полиморфный)**
- **Русский**: Полиморфный, многотипный
- **Определение**: Состояние IC, когда он видел объекты 2-4 разных форм.
- **Производительность**: Средняя, линейный поиск среди кэшированных форм
- **Степень**: Количество различных форм (2, 3, 4)

### S-T

**Speculative Optimization (Спекулятивная оптимизация)**
- **Русский**: Спекулятивная оптимизация, оптимизация на основе предположений
- **Определение**: Подход, при котором компилятор делает предположения о типах на основе наблюдений и генерирует специализированный код.
- **Защита**: Type guards проверяют предположения в runtime

**Smi (Small Integer)**
- **Русский**: Малое целое
- **Определение**: Внутреннее представление V8 для малых целых чисел (-2^30 до 2^30-1).
- **Преимущество**: Хранятся напрямую без аллокации на heap

**Temporal Locality (Временная локальность)**
- **Русский**: Временная локальность
- **Определение**: Принцип, согласно которому недавно использованные данные вероятно будут использованы снова вскоре.
- **Применение**: Основа для кэширования

**TurboFan**
- **Русский**: TurboFan (название компилятора)
- **Определение**: Оптимизирующий компилятор в V8.
- **Возможности**: Инлайнинг, специализация типов, устранение избыточности, и др.

**Type Feedback (Обратная связь по типам)**
- **Русский**: Обратная связь по типам, информация о типах
- **Определение**: Информация о типах и формах объектов, собираемая во время выполнения.
- **Источник**: Inline caches
- **Использование**: Оптимизирующий компилятор

**Type Guard / Type Check (Проверка типа)**
- **Русский**: Защита типа, проверка типа
- **Определение**: Инструкция в оптимизированном коде, которая проверяет соответствие значения ожидаемому типу/форме.
- **При несовпадении**: Деоптимизация

### U-Z

**Uninitialized IC**
- **Русский**: Неинициализированный IC
- **Определение**: Начальное состояние IC, когда он еще не выполнялся.
- **Переход**: → Monomorphic после первого выполнения

## Заключение

### Ключевые выводы

1. **Мономорфный код — оптимален**: Когда операция видит объекты только одной формы, VM может максимально оптимизировать код.

2. **Полиморфизм имеет стоимость**: Каждая дополнительная форма добавляет:
   - Проверки типов
   - Ветвления (плохо для CPU branch predictor)
   - Сложность для оптимизатора

3. **Мегаморфный код — избегайте**: Более 4 форм в одной точке → использование медленной глобальной хеш-таблицы.

4. **Скрытые классы критичны**: Понимание того, как V8 создает скрытые классы, помогает писать оптимальный код:
   - Инициализируйте все свойства
   - Используйте одинаковый порядок
   - Избегайте `delete`

5. **Профилирование > Догадок**: Не оптимизируйте преждевременно. Используйте инструменты профилирования для поиска реальных bottleneck'ов.

6. **Архитектура > Микрооптимизации**: Чистый, поддерживаемый код с умеренным полиморфизмом лучше, чем нечитаемые микрооптимизации.

### Практический совет

Следуйте правилу **"Измерь, оптимизируй, проверь"**:

1. **Измерь**: Профилируйте реальное приложение на реальных данных
2. **Оптимизируй**: Если нашли bottleneck связанный с полиморфизмом — оптимизируйте
3. **Проверь**: Убедитесь, что оптимизация действительно помогла
