GRASP: Принцип “Информационный эксперт” (Information Expert)

Обзор

Данная лекция посвящена детальному изучению принципа Information Expert (Информационный эксперт) из набора паттернов GRASP (General Responsibility Assignment Software Patterns), адаптированного для JavaScript и Node.js. Принцип определяет, какому объекту или модулю следует назначить определенную ответственность, основываясь на том, у кого есть необходимая информация для выполнения задачи.

Основные концепции

Что такое принцип “Информационный эксперт”?

Информационный эксперт — это фундаментальный принцип распределения ответственности в объектно-ориентированном проектировании. Суть принципа:

Ответственность должна быть назначена тому объекту или модулю, который обладает необходимой информацией для выполнения этой ответственности.

Ключевые положения принципа

  1. Инкапсуляция данных: Объект должен управлять своими собственными данными и не позволять внешнему коду “лезть” в свои внутренности.

  2. Сокрытие реализации: Внутренняя структура данных должна быть скрыта от внешнего мира через публичный интерфейс.

  3. Консистентность данных: Объект самостоятельно поддерживает согласованность своих данных.

  4. Независимость компонентов: Изменения внутренней реализации не должны влиять на внешний код, использующий объект.

Пример 1: Нарушение принципа со связанным списком

Проблема: Доступ к внутренней структуре данных

Рассмотрим простой связанный список с методами push и pop:

class LinkedList {
  constructor() {
    this.first = null; // Первый элемент списка
    this.length = 0;   // Длина списка
  }

  push(value) {
    // Внутренняя реализация добавления элемента
    // (детали опущены для краткости)
  }

  pop() {
    // Внутренняя реализация извлечения элемента
    // (детали опущены для краткости)
  }
}

Правильное использование (без нарушения принципа)

const list = new LinkedList();

// Используем только публичный интерфейс
list.push(1);
list.push(2);
list.push(3);

console.log(list.pop()); // 3
console.log(list.pop()); // 2
console.log(list.pop()); // 1

// Результат: все работает корректно

Нарушение принципа Information Expert

const list = new LinkedList();
list.push(1);
list.push(2);
list.push(3);

// НАРУШЕНИЕ: лезем во внутреннюю структуру!
list.last.prev.prev = list.last;

// Результат: нарушена внутренняя консистентность списка
console.log(list.pop()); // 2
console.log(list.pop()); // 2 (дублируется!)
console.log(list.pop()); // ...неправильное поведение

Проблема: Прямое обращение к внутренним полям (list.last.prev.prev) нарушает целостность внутренней структуры данных, что приводит к некорректной работе методов списка.

Пример 2: Иерархия покупок и продуктов (Composite Pattern)

Постановка задачи

Необходимо создать систему для управления покупками, где:

  • Есть отдельные продукты (Product) с ценой
  • Есть покупки (Purchase), которые могут содержать как продукты, так и другие покупки
  • Нужно вычислять общую стоимость на любом уровне иерархии

Реализация класса Product

class Product {
  constructor(name, price) {
    this.name = name;
    this.productPrice = price; // Внутреннее поле для хранения цены
  }

  // Геттер для единообразного интерфейса
  get price() {
    return this.productPrice;
  }
}

Обратите внимание:

  • Хотя можно было напрямую использовать this.price, мы создаем геттер для единообразия интерфейса с классом Purchase.
  • Продукт сам знает свою цену и возвращает её через публичный интерфейс.

Реализация класса Purchase

class Purchase {
  constructor(name, ...items) {
    this.name = name;
    this.items = items; // Коллекция продуктов и/или других покупок
  }

  // Геттер вычисляет общую стоимость, проходя по коллекции
  get price() {
    let total = 0;
    for (const item of this.items) {
      total += item.price; // Вызываем .price у каждого элемента
    }
    return total;
  }
}

Ключевые моменты:

  • Purchase не проверяет, что именно лежит в коллекции — Product или другой Purchase.
  • Благодаря единому интерфейсу (геттер price), код работает полиморфно.
  • Вложенность может быть любой: покупка внутри покупки внутри покупки.

Пример использования

// Создаем отдельные продукты
const laptop = new Product('Laptop', 1000);
const mouse = new Product('Mouse', 50);
const keyboard = new Product('Keyboard', 150);

// Группируем в покупку "Электроника"
const electronics = new Purchase('Electronics', laptop, mouse, keyboard);

// Создаем еще продукты
const bag = new Product('Bag', 80);
const mousepad = new Product('Mousepad', 20);

// Группируем в покупку "Текстиль"
const textile = new Purchase('Textile', bag, mousepad);

// Создаем общую покупку "Рабочий набор"
const workSet = new Purchase('Work Set', electronics, textile);

// Можем получить цену на любом уровне
console.log('Electronics:', electronics.price);  // 1200
console.log('Textile:', textile.price);          // 100
console.log('Total:', workSet.price);            // 1300

Визуализация иерархии

Work Set (1300)
├── Electronics (1200)
│   ├── Laptop (1000)
│   ├── Mouse (50)
│   └── Keyboard (150)
└── Textile (100)
    ├── Bag (80)
    └── Mousepad (20)

Почему это соответствует принципу Information Expert?

  1. Product сам знает свою цену и не требует от других объектов управления этой информацией.
  2. Purchase сам знает свои элементы и умеет вычислять общую стоимость, не раскрывая детали реализации.
  3. Нет необходимости во внешнем коде проверять типы элементов или обращаться к внутренним полям.
  4. Каждый объект является экспертом по своим собственным данным.

Расширяемость и гибкость

Благодаря соблюдению принципа, можно легко расширять систему:

// Можно добавить специальную логику вычисления цены
class DiscountProduct extends Product {
  constructor(name, price, discount) {
    super(name, price);
    this.discount = discount;
  }

  get price() {
    // Переопределяем вычисление цены с учетом скидки
    return this.productPrice * (1 - this.discount);
  }
}

// Можно создать специальные типы покупок
class SpecialOfferPurchase extends Purchase {
  get price() {
    // Применяем специальную скидку ко всей покупке
    const basePrice = super.price;
    return basePrice * 0.9; // 10% скидка
  }
}

Контракт и тестирование

В JavaScript нет явных интерфейсов, поэтому контракт (одинаковый геттер price) поддерживается:

  1. Силой воли разработчиков — следование соглашениям
  2. Тестами — проверка ожидаемого поведения
// Пример теста
const total = workSet.price;
console.assert(total === 1300, 'Total price should be 1300');

// Если где-то неправильно вычислена цена — тест не пройдет

Пример 3: Циклический буфер (Ring Buffer)

Назначение

Циклический буфер — это структура данных фиксированного размера, которая при переполнении начинает перезаписывать данные с начала.

Реализация на основе Buffer (Node.js)

class RingBuffer {
  constructor(size) {
    this.buffer = Buffer.alloc(size); // Выделяем буфер заданного размера
    this.offset = 0;                   // Текущее смещение для записи
    this.size = size;                  // Размер буфера (дублируем для удобства)
  }

  write(str) {
    // Получаем длину строки в байтах
    const len = Buffer.byteLength(str, 'utf8');

    // Вычисляем, сколько байтов влезет до конца буфера
    const restBuffer = this.size - this.offset;

    if (len < restBuffer) {
      // Строка полностью помещается в оставшееся место
      this.buffer.write(str, this.offset, 'utf8');
      this.offset += len;
    } else {
      // Строка не помещается, нужно разделить на две части

      // Первая часть: записываем до конца буфера
      const firstPart = str.slice(0, restBuffer);
      this.buffer.write(firstPart, this.offset, 'utf8');

      // Вторая часть: записываем с начала буфера (рекурсивно)
      this.offset = 0;
      const secondPart = str.slice(restBuffer);
      this.write(secondPart); // Рекурсивный вызов
    }
  }

  // Метод для чтения всего содержимого буфера
  toString() {
    return this.buffer.toString('utf8');
  }
}

Оптимизированная версия (использует Buffer.write)

class RingBuffer {
  constructor(size) {
    this.buffer = Buffer.alloc(size);
    this.offset = 0;
    this.size = size;
  }

  write(str) {
    const len = Buffer.byteLength(str, 'utf8');
    const restBuffer = this.size - this.offset;

    if (len < restBuffer) {
      // Используем встроенный метод buffer.write() - более эффективно
      this.buffer.write(str, this.offset, 'utf8');
      this.offset += len;
    } else {
      // Записываем первую часть
      const firstPart = str.slice(0, restBuffer);
      this.buffer.write(firstPart, this.offset, 'utf8');

      // Записываем вторую часть с начала
      this.offset = 0;
      const secondPart = str.slice(restBuffer);
      this.write(secondPart);
    }
  }

  toString() {
    return this.buffer.toString('utf8');
  }
}

Эволюция реализации:

  • Первая версия: побайтовое копирование в цикле (медленно)
  • Вторая версия: использование buffer.write() (быстро)
  • Важно: Внешний интерфейс остался неизменным!

Пример использования

const ring = new RingBuffer(10);

ring.write('1');
ring.write('2');
ring.write('3');
ring.write('4567890a');

console.log(ring.toString()); // "a234567890"

Что произошло:

  1. Записали '1' (offset = 1)
  2. Записали '2' (offset = 2)
  3. Записали '3' (offset = 3)
  4. Записали '4567890a' — строка из 8 символов:
    • В буфере 7 свободных мест (10 - 3)
    • '4567890' записалось в позиции 3-9
    • 'a' записалось в позицию 0, перезаписав '1'
  5. Результат: "a234567890"

Правильное использование (соблюдение Information Expert)

// ПРАВИЛЬНО: используем публичный интерфейс
const content = ring.toString();
console.log(content);

Нарушение принципа

// НЕПРАВИЛЬНО: лезем во внутренний буфер напрямую
ring.buffer.write('b', 0, 'utf8');
console.log(ring.buffer.toString('utf8')); // Нарушили инкапсуляцию!

Проблема: Прямое обращение к ring.buffer обходит логику управления смещением (offset), что может привести к некорректному состоянию объекта.

Исправленная версия с методом toString()

class RingBuffer {
  constructor(size) {
    this.buffer = Buffer.alloc(size);
    this.offset = 0;
    this.size = size;
  }

  write(str) {
    const len = Buffer.byteLength(str, 'utf8');
    const restBuffer = this.size - this.offset;

    if (len < restBuffer) {
      this.buffer.write(str, this.offset, 'utf8');
      this.offset += len;
    } else {
      const firstPart = str.slice(0, restBuffer);
      this.buffer.write(firstPart, this.offset, 'utf8');
      this.offset = 0;
      const secondPart = str.slice(restBuffer);
      this.write(secondPart);
    }
  }

  // Добавили метод для чтения — инкапсулируем доступ к buffer
  toString() {
    return this.buffer.toString('utf8');
  }
}

// Использование
const ring = new RingBuffer(10);
ring.write('1');
ring.write('2');
ring.write('3');
ring.write('4567890ab');

// Правильно: используем публичный метод
console.log(ring.toString()); // "ab34567890"

Пример 4: Процедурный подход с модулями

От ООП к процедурному программированию

Принцип Information Expert применим не только к классам, но и к модулям. Рассмотрим процедурную версию циклического буфера.

Модуль ring.js (процедурный стиль)

// ring.js - модуль для работы с циклическим буфером

/**
 * Создает циклический буфер заданного размера
 * @param {number} size - размер буфера в байтах
 * @returns {Object} объект с полями buffer, offset, size
 */
const alloc = (size) => ({
  buffer: Buffer.alloc(size),
  offset: 0,
  size
});

/**
 * Записывает строку в циклический буфер
 * @param {Object} buf - объект буфера, созданный через alloc()
 * @param {string} str - строка для записи
 */
const write = (buf, str) => {
  const len = Buffer.byteLength(str, 'utf8');
  const restBuffer = buf.size - buf.offset;

  if (len < restBuffer) {
    buf.buffer.write(str, buf.offset, 'utf8');
    buf.offset += len;
  } else {
    const firstPart = str.slice(0, restBuffer);
    buf.buffer.write(firstPart, buf.offset, 'utf8');
    buf.offset = 0;
    const secondPart = str.slice(restBuffer);
    write(buf, secondPart); // Рекурсивный вызов
  }
};

/**
 * Читает содержимое буфера как строку
 * @param {Object} buf - объект буфера
 * @returns {string} содержимое буфера в виде строки
 */
const read = (buf) => {
  return buf.buffer.toString('utf8');
};

// Экспортируем публичный интерфейс модуля
module.exports = { alloc, write, read };

Использование модуля

// main.js
const ring = require('./ring');

// Создаем буфер
const buf = ring.alloc(10);

// Записываем данные
ring.write(buf, '1');
ring.write(buf, '2');
ring.write(buf, '3');
ring.write(buf, '4567890a');

// Читаем содержимое
console.log(ring.read(buf)); // "a234567890"

// Дополнительные записи
ring.write(buf, 'b');
console.log(ring.read(buf)); // "ab34567890"

ring.write(buf, '***');
console.log(ring.read(buf)); // "***4567***"
// (звездочки перезаписали часть данных циклически)

Почему это Information Expert на уровне модуля?

  1. Модуль инкапсулирует логику: Все функции для работы с циклическим буфером находятся в одном месте.

  2. Единственный эксперт: Только модуль ring знает, как правильно работать с внутренней структурой буфера.

  3. Публичный интерфейс: alloc, write, read — это контракт модуля, который не зависит от внутренней реализации.

  4. Возможность изменения реализации: Внутреннюю логику можно оптимизировать без изменения интерфейса.

Вывод: Принцип Information Expert работает в обоих стилях программирования!

Ключевые выводы

1. Определение ответственности

Принцип: Назначайте ответственность объекту (модулю), который владеет необходимой информацией.

// ПЛОХО: внешний код знает о внутренней структуре
function calculateTotal(purchase) {
  let total = 0;
  for (const item of purchase.items) {
    if (item instanceof Product) {
      total += item.productPrice; // Лезем в приватное поле
    } else if (item instanceof Purchase) {
      total += calculateTotal(item); // Рекурсия
    }
  }
  return total;
}

// ХОРОШО: объект сам знает, как вычислить свою цену
class Purchase {
  get price() {
    return this.items.reduce((sum, item) => sum + item.price, 0);
  }
}

2. Инкапсуляция защищает от ошибок

Нарушение инкапсуляции приводит к:

  • Нарушению консистентности данных
  • Хрупкому коду (изменения внутренней структуры ломают внешний код)
  • Сложности в поддержке и тестировании

3. Единый интерфейс для разных типов

Благодаря единому интерфейсу (например, геттер price), код работает полиморфно:

// Не важно, Product это или Purchase — интерфейс одинаковый
items.forEach(item => console.log(item.price));

4. Тестирование вместо явных интерфейсов

В JavaScript нет встроенных интерфейсов, поэтому контракты обеспечиваются через:

  • Соглашения (naming conventions)
  • Тесты (проверка ожидаемого поведения)
  • Документацию (JSDoc, комментарии)
/**
 * Контракт: все элементы должны иметь геттер price
 * @interface PricedItem
 * @property {number} price - цена элемента
 */

5. Гибкость и расширяемость

Правильное применение принципа позволяет:

  • Менять внутреннюю реализацию без изменения интерфейса
  • Легко добавлять новые типы объектов
  • Минимизировать связанность между компонентами

6. Применимость на разных уровнях

Information Expert работает на уровне:

  • Классов и объектов (ООП)
  • Модулей (процедурное программирование)
  • Микросервисов (архитектура)

Практические рекомендации

✅ Правильно

// 1. Объект управляет своими данными
class User {
  constructor(name, age) {
    this._name = name;
    this._age = age;
  }

  get isAdult() {
    return this._age >= 18; // Сам знает, как проверить
  }
}

// 2. Используем публичный интерфейс
const user = new User('Alice', 25);
console.log(user.isAdult); // true

// 3. Модуль инкапсулирует логику
const dateUtils = {
  parse: (str) => { /* ... */ },
  format: (date) => { /* ... */ }
};

❌ Неправильно

// 1. Проверка в внешнем коде
function checkAdult(user) {
  return user._age >= 18; // Лезем в приватное поле!
}

// 2. Прямой доступ к внутренним данным
list.first.next.next = null; // Нарушаем консистентность!

// 3. Обход инкапсуляции
ring.buffer.write('x', 0); // Минуем логику ring.write()

Контрольные вопросы при проектировании

Задавайте себе эти вопросы при распределении ответственности:

  1. Кто владеет данными? → Он должен управлять ими
  2. Кому нужна эта информация? → Данные должны быть рядом
  3. Может ли кто-то снаружи нарушить консистентность? → Нужна инкапсуляция
  4. Можно ли изменить реализацию, не трогая интерфейс? → Хорошая инкапсуляция

Связь с другими принципами

Information Expert и SOLID

  • Single Responsibility: Information Expert помогает определить единственную ответственность объекта
  • Open/Closed: Инкапсуляция позволяет расширять поведение без изменения существующего кода
  • Liskov Substitution: Единый интерфейс обеспечивает взаимозаменяемость объектов

Information Expert и другие GRASP-паттерны

  • Low Coupling: Инкапсуляция снижает связанность между компонентами
  • High Cohesion: Данные и операции над ними находятся в одном месте
  • Creator: Тот, кто создает объект, часто является Information Expert по его инициализации

Заключение

Принцип Information Expert — это фундаментальный паттерн, который помогает правильно распределять ответственность в программных системах. Следование этому принципу приводит к созданию:

  • 🎯 Слабосвязанного кода — компоненты независимы друг от друга
  • 🔒 Надежной инкапсуляции — внутренние данные защищены от некорректного использования
  • 🔧 Поддерживаемого кода — изменения локализованы внутри компонентов
  • 🧪 Тестируемого кода — публичный интерфейс легко тестировать
  • 📈 Расширяемой архитектуры — новый функционал добавляется без изменения существующего

Применяйте Information Expert на всех уровнях: от отдельных классов до модулей и микросервисов. Помните главное правило: ответственность принадлежит тому, кто владеет информацией.
